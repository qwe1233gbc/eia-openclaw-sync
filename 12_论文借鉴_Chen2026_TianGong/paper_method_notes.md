# Chen 2026 论文思路与本课题迁移方案

## 1. 论文研究问题

论文指出环境领域 LLM 部署存在两个瓶颈：

- 结构化领域知识不足：通用模型虽然语言能力强，但在法规、机理、核算、验证等任务中容易缺少稳定的领域依据。
- 部署策略不清：不同环境决策任务需要不同的知识组织方式，有些更依赖检索、工具调用、计算和动态推理。

对本课题而言，最可迁移的部分是“分层部署”：稳定任务沉淀为结构化知识和审核 skill，复杂动态任务使用通用模型加检索、工具和人工复核。

## 2. 数据集构建思路

论文的数据构建流程可以概括为：

1. 汇集权威教材语料。
2. 对教材进行 OCR、切块、向量化，构建向量数据库。
3. 同步构建全文检索库，支持关键词检索。
4. 使用种子问题在向量库和全文库中混合召回片段。
5. 去重并扩展上下文。
6. 使用 LLM 按固定 schema 生成结构化问题、答案和推理说明。
7. 由专家核对生成答案是否被原文证据支持。
8. 删除幻觉、证据不匹配或题文不一致的条目。

## 3. 评测与部署思路

论文比较了不同模型和 agentic workflow 的效果。对本课题最重要的结论是：

- agentic workflow 在多步骤、跨材料、需要计算或工具辅助的任务上更灵活。
- 稳定、标准化任务适合沉淀为结构化知识、规则和审核 skill。
- 复杂审核任务优先使用通用模型 + 检索 + 工具 + 人工复核。

## 4. 对本课题的可迁移设计

| Chen 2026 模块 | 本课题对应模块 | 迁移方式 |
| --- | --- | --- |
| 教材语料库 | 佛山市塑胶指南、法规标准、Dify 知识库、样本链、修改意见 | 统一切块并保留来源、章节、证据等级 |
| Pinecone 向量库 | 本目录 SQLite 向量表；后续可替换为 Pinecone/FAISS/Chroma | 先本地验证召回效果，再迁移服务化 |
| OpenSearch 全文库 | SQLite FTS5 | 支持标准号、项目名称、污染物名称等关键词检索 |
| 种子问题 | 审核任务样本 Q01-Q18、Dify 工作流规则、人工审核问题 | 用种子问题驱动证据召回和任务扩展 |
| 结构化生成 | 审核问答、审核任务三元组、建议修改意见 | 使用 JSON schema 约束输出字段 |
| 专家复核 | 人工复核结论、金标答案校准 | 标注证据是否充分、是否过度泛化 |
| 分层部署策略 | Dify / Python 审核工作流 + 审核技能库 | 稳定规则任务沉淀为 skill，复杂任务保持检索和人工复核 |

## 5. 本课题应优先复用的模块

1. 混合检索：法规/标准依据、审核技能、类案经验和样本证据必须同时支持语义召回和关键词召回。
2. 证据可追溯 schema：每个 chunk 必须记录原文件、章节、页码/行号、证据类型。
3. 种子问题驱动的数据扩展：从已有审核任务出发，召回材料后生成候选审核问答。
4. 结构化输出：问题、答案、证据、标准依据、类案经验、人工复核状态必须分字段。
5. 专家复核闭环：生成任务不能直接作为金标，应先进入待复核状态。

## 6. 本课题不宜直接照搬的部分

- 不宜直接照搬其教材问答数据，因为本课题聚焦环评审核，知识来源应以标准、指南、报告和修改意见为主。
- 不宜把论文中的模型路线作为本课题目标；本课题重点是法规/标准依据库、审核技能库和证据可追溯评价。
- 不宜把复杂审核任务简化为单轮问答，应保留检索、计算、证据校验和人工复核节点。
- 不宜直接复制外部仓库大段代码；本目录使用本仓库场景重写轻量模板，并保留来源说明。
